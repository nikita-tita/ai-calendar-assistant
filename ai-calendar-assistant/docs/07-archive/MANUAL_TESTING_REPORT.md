### 1.2. Создание событий

#### 1.2.1 Создание одного события

**Сценарий 1: Событие с полной информацией**
```
Пользователь → "Встреча с клиентом Ивановым завтра в 15:00"
```

**Ожидаемое поведение:**
1. LLM распознает intent: `create`
2. Извлекает:
   - Название: "Встреча с клиентом Ивановым"
   - Дата: завтра (относительно текущей даты)
   - Время: 15:00
   - Длительность: 60 минут (по умолчанию)
3. Создает событие в Radicale
4. Отправляет подтверждение с деталями

**Сценарий 2: Событие с неполной информацией**
```
Пользователь → "Встреча с клиентом"
```

**Ожидаемое поведение:**
1. LLM распознает intent: `clarify`
2. Запрашивает недостающую информацию:
   - "Когда запланировать встречу?"
   - "В какое время?"
3. Использует контекст предыдущих сообщений

**Сценарий 3: Событие с длительностью**
```
Пользователь → "Показ квартиры послезавтра в 14:00 на 2 часа"
```

**Ожидаемое поведение:**
1. Название: "Показ квартиры"
2. Время начала: послезавтра 14:00
3. Длительность: 120 минут
4. Время окончания: 16:00

**Статус:** ✅ Функция реализована
**Код:** [app/services/llm_agent_yandex.py:38-50](app/services/llm_agent_yandex.py#L38-L50)
**LLM модель:** Yandex GPT (работает без VPN)

#### 1.2.2 Создание нескольких событий сразу

**Сценарий 1: Batch-создание (расписание на день)**
```
Пользователь → "тайминг на 23 октября:
12:45-13:00 Приезд, заселение
13:00-13:30 Кофе-брейк
13:30-14:30 Воркшоп по продукту
14:30-15:00 Перерыв
15:00-16:30 Нетворкинг"
```

**Ожидаемое поведение:**
1. LLM распознает формат расписания (3+ строк с временными диапазонами)
2. Intent: `batch_create`
3. Создает массив событий (5 событий)
4. Показывает превью всех событий
5. Запрашивает подтверждение
6. Создает все события одновременно

**Сценарий 2: Перечисление событий**
```
Пользователь → "Создай мне встречи: с Петровым завтра в 10:00, с Ивановым в 14:00 и с Сидоровым послезавтра в 11:00"
```

**Ожидаемое поведение:**
1. Распознает 3 отдельных события
2. Создает batch_actions массив
3. Показывает список всех событий
4. Запрашивает подтверждение
5. Создает все 3 события

**Статус:** ✅ Функция реализована
**Код:** [app/services/llm_agent_yandex.py:91-115](app/services/llm_agent_yandex.py#L91-L115)
**Особенность:** Использует формат HH:MM-HH:MM для распознавания расписаний

#### 1.2.3 Создание повторяющихся событий (recurring)

**Сценарий 1: Ежедневное событие без ограничения**
```
Пользователь → "Каждый день в 9:00 утренняя планерка"
```

**Ожидаемое поведение:**
1. Intent: `create_recurring`
2. Параметры:
   - recurrence_type: "daily"
   - start_time: 09:00
   - title: "утренняя планерка"
   - recurrence_end_date: конец текущего года
3. Показывает информацию о количестве событий
4. Запрашивает подтверждение
5. Создает все события до конца года

**Сценарий 2: Ежедневное событие на период**
```
Пользователь → "Зарядка каждый день в 7:00 на неделю"
```

**Ожидаемое поведение:**
1. Intent: `create_recurring`
2. recurrence_end_date: сегодня + 7 дней
3. Создает 7 событий

**Сценарий 3: Еженедельное событие (определенные дни)**
```
Пользователь → "Каждый понедельник и среду в 18:00 тренировка"
```

**Ожидаемое поведение:**
1. Intent: `create_recurring`
2. recurrence_type: "weekly"
3. recurrence_days: ["mon", "wed"]
4. Создает события на все понедельники и среды до конца года

**Статус:** ✅ Функция реализована
**Код:** [app/services/llm_agent_yandex.py:60-81](app/services/llm_agent_yandex.py#L60-L81)
**Поддерживаемые типы:** daily, weekly, monthly

## 5. Обнаруженные проблемы и рекомендации

### 5.1. Критические

1. **Бот недвижимости не развернут**
   - Приоритет: ВЫСОКИЙ
   - Действие: Развернуть docker-compose.property.yml
   - Требуется: PostgreSQL, настройка Yandex API keys

2. **Веб-приложение не протестировано**
   - Приоритет: ВЫСОКИЙ
   - Действие: Проверить DNS домена, SSL сертификат
   - Проверить доступность из Telegram WebApp

---

### 5.2. Важные

3. **Yandex GPT API keys не настроены**
   - В .env: `YOUR_YANDEX_GPT_API_KEY_HERE`
   - Бот календарь работает, но без LLM не будет интеллектуальных функций
   - Действие: Настроить реальные ключи

4. **Напоминания могут не работать**
   - Daily reminders запускаются в background
   - Требует проверки логов: `docker logs telegram-bot-polling | grep reminder`

5. **Webhook не используется**
   - Бот работает в polling режиме (менее эффективно)
   - Рекомендация: Настроить webhook для production

---

### 5.3. Улучшения

6. **Хранилище conversation_history в памяти**
   - Теряется при перезапуске контейнера
   - Рекомендация: Использовать Redis или файловое хранилище

7. **User timezones в памяти**
   - Не персистентны
   - Рекомендация: Сохранять в базу данных или файл

8. **Тестовое покрытие**
   - Требуется больше автотестов
   - Особенно для LLM агента и парсинга дат
